Hello Git

What is Git? Git is a type of versioning software that allows us to keep track of changes we make to a project as we build it. Git records changes you make to an project, stores those chanages, then allows you to reference the changes as needed.

Hold on, what actually is versioning or "version control", and why do we use it in software development? Version control is akin to creating different drrafts of a story. You wold not write an entire story in one sitting. Instead, you would write multiple drafts, makes changes along the way, keep final versions intact, and incorporate changes as needed. In software development, version control allows you yo create different versions of your code as you complete each milestone.

git init

git init

Keyword init stands for initialize a local Git repo (aka repository) on your own computer. A Repository or repo refers to a directory where we store the versions of our code. As you create versions of your project, your local repo will keep track of each new version.

The git init command sets up the Git repo and tools Git needs to track changes to the project.

Once you initialize a new Git repo with git init your terminal will tell you where the new repo was created:

Initalized an empty git repository in /home/ccuser/workspace/ready-sorcerers-code/.git/

git Workflow

A Git project can be thought of as having 3 parts:

  1. A Working Directory: where you'll be doing all the work: creating, editing, deleting, and organizing files
  2. A Staging Area: where you'll list changes you make to the working directory
  3. A Repository: where Git permanently stores those changes as different versions of the project

Git workflow consists of editing files in the working directory on your computer, adding files to the staging area, and saving changes to a Git Repository. In Git we save changes with a commit

git status

As you update & change the contents of your documents you will be making those changes in the working directory. You can check the status of the changes that you make with:

git status

git status command will print a log of the changes that have occurred in the current version of a project.

git add

Git doesn't know about files that it should track until they have been added to the staging area. To add files to be tracked to the staging area we use git add filename. filename here refers to the the name of the file that you are editing that you want moved the the staging area for git to track changes.

To speed up your workflow if you have to add multiple files in a project to a staging area at once you can modify the git add command to git add ., which will add all the files & directories in the current directory. So, if we enter the git add . from the root directory of a project, we add all the files & directories of a project to the staging area at once!

git commit

A commit is the final step in our Git workflow. A commit permanently stores changes from the staging area inside the repository.

git commit commits all the files in your staging area. Hwever, one more piece of code is needed for a commit: the option -m followed by a message. For example:

git commit -m "Complete first line of dialogue."

A commit message starts with an imperative verb and ends with a period, for example "Fix broken registration link."

Good commit messages are written:

  1. in quotation marks
  2. in present tense
  3. in 50 character or less using -m

After committing, we can check the status of the project and see that we have no new changes. This means we created a new version of our project, and are now comparing any changes to that version.

Create a Remote Repo[sitory] on GitHub

Often times you're going to be storing your code to a remote repo better known as a repository hosted on the internet or other network. This will be an identical repository, but stored in the cloud, so that anything happens to your computer or you want your code to be accessible to other, it'll be online!

GitHub is one such service for hosting remote repos on the web. Once the repo is stored online, you can make Git repos open source, track tasks & issues, manage projects, and collaborate with other!

git remote add

Now Git, your local repo needs to know about the repo you created on GitHub (aka the remote repo) so that you can store your project there.

To specify the remote GitHub repo, we add the remote and label it as the origin as such

git remote add origin https://github.com/your-username/your-repository-name
  1. The remote is the URL of the repository that will store your project's contents.
  2. The origin is an alias for the remote repo. This means that instead of typing the lengthy remote GitHub URL each time, you can now just refer to it as origin
  3. The URL
    https://github.com/your-username/your-repository-name
    is the remote URL that refers to the repo youh created on GitHub earlier.
    • Replace your-username with your GitHub username.
    • Replace your-repository-name with thr GitHub repo's name you created

You can confirm the remote has been added with the following:

git remote -v

git push upstream

To push new versions (commits) to our remote repo on GitHub all you need is this command:

git push -u origin master

Let's break down this command:

  1. git push informs Git that we want to update a remote repo. In other words, we want to 'push' changes from a local repo to a remote repo
  2. origin refers to the remote repo that we are pushing to on GitHub.
  3. master refers to the place where our work lives on our local repo
  4. -u stands for "upstream". By writing the command with the upstream option, we establish a link between the local repo & the remote master branch. Git now knows which remote repo to push our changes to!

An important thing to note, we only need to user -u option the first time we push to the remote repo.

git push

Once you connect the local and remote repos you can pushc changes using git push origin master you non longer need the -u for upstream Using this command backs up our code to GitHub & it's good practice to make sure that you push after every commit to make sure not to lose any work.

Git is the industry-standard version control system for web developers & you have the basic workflow down! Let's break down what we covered for Git:

Use Git commands to help keep track of changes made to a project:

  • git init creates a new Git repository.
  • git status inspects the contents of the working directory and staging area.
  • git add filename adds files from the working directory to the staging area.
  • git add . adds all files from the working directory to the staging area.
  • git commit -m "commit message" permanently stores file changes from the staging area in the repository.

GitHub is a service for hosting remote repositories on the web.

  • git remote add origin url specifies the remote repository using Git
  • git push -u origin master pushes the changes to the master branch on the remote repository, linking the local repository to the remote repository.
  • git push origin master pushes the changes to the master branch on the remote repository, given that the local repository and the remote repository are already linked.

Differences between Git & GitHub

Git is a widely used version control system used to manage code. Git allows you to save drafts of your code so that you cna look back at previous versions & potentially undo complicated errors. A project managed with Git is called a Git repository.

GitHub is a popular hosting service for Git Repos. GitHub allows you to store your local Git Repos in the cloud. With GitHub, you can backup your personal files, share your code, & collab with other devs.
